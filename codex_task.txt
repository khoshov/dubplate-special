Файлы и пути:
- Админка Records: dubplate-special/apps/records/admin/
- VK сервис: dubplate-special/apps/records/services/social/vk_service.py

Задача: сделать 2 admin actions: мгновенная публикация в VK и планирование равномерно по интервалу (через промежуточную страницу). Плюс добавить автотесты новой логики.

1) VKService: два режима — мгновенно по умолчанию и отложенно по явному параметру
- Сейчас _wall_post всегда выставляет publish_date = now + POSTPONE_DELAY_SECONDS (всё уходит в отложку).
- Нужно изменить:
  - по умолчанию публиковать сразу: НЕ передавать publish_date вообще.
  - добавить поддержку отложки:
    - в `post_record_with_audio(...)` добавить именованный параметр `publish_at: datetime | None = None`
    - прокинуть до `_wall_post(..., publish_date_ts: int | None = None)`
    - в `_wall_post` добавлять `publish_date` в params только если `publish_date_ts` не None (и равен int(publish_at.timestamp())).
- Старый action “мгновенная публикация” должен вызывать `post_record_with_audio(record)` без `publish_at`.

2) Django admin: два actions в RecordAdmin
a) Action 1: "Опубликовать в VK (сразу)"
- Публикует выбранные записи сразу, без отложки.
- Ошибки VK API не должны валить весь процесс — собрать ok/fail и показать через messages.

b) Action 2: "Запланировать публикацию в VK (равномерно)"
- Action не публикует, а редиректит на отдельную промежуточную страницу (custom admin view) для выбора диапазона времени и подтверждения.
- Промежуточная страница:
  - показать список выбранных записей (минимум: id + __str__)
  - форма: publish_from + publish_to (datetime)
  - при POST:
    - валидировать: N>0, publish_to >= publish_from
    - упорядочить записи детерминированно (используй ordering админки: RecordAdmin.get_ordering(request) / self.get_ordering(request))
    - равномерно распределить времена включая границы:
      - если N=1 => publish_from
      - если N>1 => step=(to-from)/(N-1), t_i=from+step*i
    - для каждой записи вызвать `vk_service.post_record_with_audio(record, publish_at=t_i)`
    - собрать отчет ok/fail и показать через messages; после выполнения редирект обратно в changelist

3) URLs / админка
- Добавь URL для промежуточной страницы в RecordAdmin через get_urls():
  - path("vk-schedule/", self.admin_site.admin_view(self.vk_schedule_view), name="records_record_vk_schedule")
- Важно: если есть миксин, который переопределяет get_urls некорректно (например вызывает ModelAdmin.get_urls(self) напрямую), исправь на cooperative: `base_urls = super().get_urls()` чтобы не поломать существующие кастомные урлы (redeye refresh и т.п.).

  - Шаблон расширяет стандартный admin base, содержит CSRF, форму datetime, список выбранных объектов и кнопки Submit/Cancel.

5) Автотесты (pytest)
Сделай минимальные тесты, не требующие реального VK API:
A) Чистая функция расписания
- Вынеси расчет равномерного расписания в чистую функцию, например:
  - dubplate-special/apps/records/services/schedule.py -> build_even_schedule(start_at, end_at, n) -> list[datetime]
- Покрыть тестами:
  - N=1: возвращает [start_at]
  - N=5 и интервал 8 часов: дает времена с шагом 2 часа, включая end_at
  - N=2: [start_at, end_at]

B) VKService: publish_date только когда явно задан
- Замокай/перехвати место, где формируются params для wall.post (или метод, который отправляет запрос), и проверь:
  - publish_at=None => params НЕ содержат publish_date
  - publish_at задан => params содержит publish_date с ожидаемым timestamp

C) Admin view POST (точечно, без HTML снапшотов)
- Используй Django admin client / RequestFactory:
  - Авторизуй superuser, сформируй POST на vk-schedule view с ids и диапазоном времени.
  - Замокай `vk_service.post_record_with_audio`, чтобы не трогать сеть.
  - Проверь, что он вызван N раз и publish_at передан равномерно рассчитанный.

Размести тесты в `tests/` (например `tests/test_vk_schedule.py`), придерживайся pytest-django.
