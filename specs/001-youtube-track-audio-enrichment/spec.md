## User Scenarios & Testing *(mandatory)*

### User Story 1 - Автообогащение Discogs треков (Priority: P1)

Администратор добавляет запись из Discogs или запускает обновление существующей записи из Discogs и получает треки с локальными mp3-превью, если для треков доступны валидные YouTube-ссылки.

**Why this priority**: Это основной бизнес-результат фичи: восстановить аудио-обогащение именно для потока Discogs без затрагивания Redeye-потока.

**Independent Test**: Сценарий полностью проверяется через создание/обновление записи Discogs и последующую проверку, что у подходящих треков заполнен `audio_preview`, а Redeye-функции работают как прежде.

**Acceptance Scenarios**:

1. **Given** новая запись импортируется из Discogs и содержит треки с валидными YouTube-ссылками, **When** импорт завершён, **Then** для подходящих треков создаются локальные mp3-превью.
2. **Given** существующая запись обновляется через admin action "Обновить из Discogs", **When** обновление завершается, **Then** треки обновляются и дополнительно выполняется обогащение аудио из YouTube.
3. **Given** часть треков не может быть обогащена, **When** выполняется операция, **Then** остальные треки продолжают обрабатываться без остановки всей операции.

---

### User Story 2 - Массовое ручное обновление YouTube-аудио (Priority: P2)

Администратор выбирает несколько записей в списке админки и запускает отдельный admin action "Обновить аудио из YouTube", чтобы принудительно перезаписать mp3-превью.

**Why this priority**: Нужен управляемый массовый путь поддержки каталога, независимый от полного обновления данных из внешнего источника.

**Independent Test**: Сценарий проверяется отдельно от Discogs-обновления: выбрать записи в списке, запустить action, убедиться, что задачи поставлены в очередь, и проверить детальный итог в отчёте задачи.

**Acceptance Scenarios**:

1. **Given** выбраны записи с уже существующими `audio_preview`, **When** администратор запускает "Обновить аудио из YouTube", **Then** интерфейс сразу подтверждает запуск и создаётся отчёт задачи.
2. **Given** задачи завершены, **When** администратор открывает отчёт, **Then** он видит `updated/skipped/errors` и причины пропусков/ошибок.
3. **Given** по записи уже выполняется активная задача, **When** администратор запускает новую операцию для той же записи, **Then** повторный запуск для этой записи помечается как `skipped (already running)`.

---

### User Story 3 - Точечное обновление с формы записи (Priority: P3)

Администратор на странице редактирования записи запускает кнопку для обновления аудио треков через YouTube-модуль и получает прозрачный итог по обновлённым, пропущенным и ошибочным трекам.

**Why this priority**: Это быстрый точечный инструмент для ручной коррекции конкретной записи без массовых операций.

**Independent Test**: Сценарий тестируется отдельно через форму редактирования одной записи: нажатие кнопки, выполнение операции и проверка итогового сообщения.

**Acceptance Scenarios**:

1. **Given** администратор находится на странице редактирования записи, **When** запускает кнопку обновления аудио из YouTube, **Then** обрабатываются треки только этой записи.
2. **Given** во время обработки есть частичные ошибки, **When** администратор запускает операцию, **Then** интерфейс сразу подтверждает постановку в очередь и показывает ссылку на отчёт задачи; после завершения задачи администратор видит в отчёте суммарный результат (`updated/skipped/errors`) и причины.

---

### Edge Cases

- У трека отсутствует `youtube_url`.
- У трека `youtube_url` есть, но ссылка невалидна или недоступна.
- Видео найдено, но совпадение по названию трека и/или артистам ниже допустимого порога.
- У трека уже есть `audio_preview`, и требуется принудительная перезапись.
- По части треков возникают ошибки сети или ограничений внешнего источника, но остальные треки должны обработаться.
- В одной операции встречаются одновременно успешные, пропущенные и ошибочные треки; отчёт должен оставаться полным и читаемым.
- Для одной и той же записи одновременно инициируются несколько запусков из разных `admin action`.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST подтягивать локальные mp3-превью к трекам записей, импортированных из Discogs, если для треков доступны валидные источники YouTube-аудио.
- **FR-002**: Система MUST использовать текущую структуру треков (`audio_preview`, `youtube_url`) и не нарушать существующую докачку аудио из Redeye.
- **FR-003**: Система MUST встроить докачку аудио из YouTube в существующий admin action `update_from_discogs` без изменения его базового назначения (полное обновление данных записи из Discogs).
- **FR-004**: Система MUST предоставить отдельный admin action "Обновить аудио из YouTube" для выбранных записей в списке админки, который всегда перезаписывает существующие mp3.
- **FR-005**: Система MUST предоставить кнопку на странице редактирования записи в админке для запуска точечного обновления аудио треков через YouTube-модуль.
- **FR-006**: Система MUST показывать по ручным операциям понятный итог: сразу после запуска подтверждать постановку в очередь, а детальный итог (`updated/skipped/errors` с причинами) публиковать в отчёте задачи.
- **FR-007**: Система MUST обрабатывать ошибки по трекам изолированно: сбой одного трека не останавливает обработку остальных треков и записей в текущем запуске.
- **FR-008**: Система MUST пропускать треки без валидного источника (`youtube_url` отсутствует/невалиден/недоступен) и фиксировать причину пропуска в итоговом отчёте.
- **FR-014**: Система MUST применять консервативный матчинг: аудио обновляется только при совпадении названия трека и не менее одного артиста; иначе трек помечается как `skipped` с причиной `mismatch`.
- **FR-009**: Система MUST выполнять обогащение аудио для Discogs-треков в двух точках: при добавлении записи из Discogs и при обновлении через admin action "Обновить из Discogs".
- **FR-010**: Перед реализацией MUST быть выполнен и зафиксирован сравнительный анализ текущего кода и `origin/feature/add_track_dl` с тремя результатами: переиспользуемые части, несовместимости, нужные изменения.
- **FR-011**: Спецификация MUST явно фиксировать решение по очередям: в текущем релизе использовать асинхронные задачи через Celery/Redis для YouTube-аудио-обогащения.
- **FR-012**: Существующий CLI-поток `redeye_mp3_attach` MUST оставаться неизменным по назначению и совместимости.
- **FR-013**: Для каждого ручного запуска YouTube-обогащения система MUST создавать отдельную сущность отчёта задачи, доступную из админки.
- **FR-015**: Система MUST выполнять до 3 попыток обработки трека при временных ошибках внешнего источника с экспоненциальной паузой; после исчерпания попыток трек помечается как `failed`.
- **FR-016**: Система MUST запрещать параллельные активные задачи YouTube-обогащения для одной и той же записи; повторный запуск для записи в процессе отмечается как `skipped (already running)` в отчёте.

### Key Entities *(include if feature involves data)*

- **Record**: Сущность релиза; содержит источники Discogs/Redeye и набор треков для обогащения.
- **Track**: Сущность трека записи; ключевые атрибуты для фичи: `title`, `position_index`, `youtube_url`, `audio_preview`.
- **Audio Enrichment Operation Result**: Итог выполнения одной операции (массовой или точечной) с агрегатами `updated_count`, `skipped_count`, `error_count` и перечнями причин.
- **Audio Enrichment Job Report**: Отдельный отчёт асинхронного запуска с идентификатором, статусом выполнения и деталями результата по трекам/записям.
- **Legacy Comparison Report**: Документ перед реализацией, фиксирующий переиспользование legacy-идей, несовместимости и целевые изменения.

## Assumptions & Constraints *(mandatory)*

### Assumptions

- **A-001**: Источником для YouTube-аудио в рамках этой фичи является `Track.youtube_url`; отдельный процесс дополнительного сопоставления URL вне текущего scope не используется.
- **A-002**: Ручные операции выполняются пользователями админки с правами изменения записей.
- **A-003**: Для первого релиза ручные и массовые операции допускают асинхронное выполнение через очередь с отложенным завершением.
- **A-004**: Требуется предсказуемая перезапись `audio_preview` только в режимах, где это явно заявлено.

### Constraints

- **C-001**: Фича не должна менять назначение существующих потоков Redeye: `update_from_redeye`, кнопка "Закачать mp3 с Redeye", CLI `redeye_mp3_attach`.
- **C-002**: В рамках этой фичи вводится инфраструктура Celery/Redis для фонового выполнения YouTube-аудио-обогащения и управляемых retry.
- **C-003**: В интерфейсе и документации используется термин `admin action` (кроме явно существующей CLI-команды `redeye_mp3_attach`).
- **C-004**: Изменения данных должны быть недеструктивными: удаление/перезапись аудио допускается только в рамках целевого трека и только по правилам конкретной операции.
- **C-005**: Если потребуются изменения модели, они должны быть обратимо мигрируемыми и совместимыми с существующими данными треков.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: На контрольной выборке минимум из 30 Discogs-записей не менее 95% треков с валидным источником получают `audio_preview` как при добавлении записи, так и при обновлении через admin action "Обновить из Discogs".
- **SC-002**: Для обеих ручных операций (массовой и точечной) в 100% запусков интерфейс сразу подтверждает постановку в очередь, а итоговый отчёт с `updated`, `skipped`, `errors` доступен после завершения задач.
- **SC-003**: В admin action "Обновить аудио из YouTube" для треков с валидным источником существующие mp3 перезаписываются в 100% случаев, когда трек попал в обработку.
- **SC-004**: До старта реализации утверждена и зафиксирована схема фоновой обработки на Celery/Redis, включая ограничения, правила ретраев и критерии завершения задач.
- **SC-005**: При частичных сбоях в 100% запусков обработка продолжается для оставшихся треков, а в отчёте для каждого пропуска/ошибки указана причина.
- **SC-006**: Для 100% треков, помеченных как `skipped (mismatch)`, в отчёте зафиксировано, какое условие строгого матчинга не выполнено (название, артист или оба).
- **SC-007**: Для 100% треков, завершившихся со статусом `failed`, в отчёте зафиксированы количество попыток (не более 3) и финальная причина ошибки.
- **SC-008**: В 100% случаев повторного запуска по записи с активной задачей новая параллельная задача не создаётся, а запись получает статус `skipped (already running)`.

## Comparative Analysis

### Current Branch Findings

- Текущий поток Discogs обновляет данные записи и треклист, но не выполняет автоматическое YouTube-аудио-обогащение.
- Слой аудио-сервисов уже отделён от бизнес-потока записи, но сейчас покрывает только Redeye-аудио.
- Актуальная модель трека поддерживает связку `youtube_url` + локальное `audio_preview`, что позволяет встроить YouTube-обогащение без смены основной модели.
- В админке уже есть массовые и точечные ручные операции с итоговыми сообщениями, но они ориентированы на Redeye-процесс.
- На момент сравнительного анализа (до реализации этой фичи) в текущем окружении отсутствовала активная очередь фоновых задач; операционный путь был синхронным.

### Legacy Branch Findings (`origin/feature/add_track_dl`)

- Legacy-поток реализовывал YouTube-докачку как фоновую задачу с предварительным матчингом видео и последующей загрузкой аудио.
- Legacy-архитектура опиралась на очередь задач и отдельную инфраструктуру выполнения.
- Legacy-модель трека использовала другое файловое поле, поэтому прямое переиспользование кода невозможно.
- Практика запуска аудио-обработки после фиксации транзакции и изоляции ошибок по трекам является полезной для адаптации.

### Reuse, Incompatibilities, Required Changes

- **Переиспользуемые идеи legacy**: проверка релевантности видео к треку (track-title/artist match), изоляция ошибок по трекам, запуск обработки после фиксации транзакции.
- **Несовместимости**: старая модель `Track.file` против текущей `Track.audio_preview`; legacy-асинхронность на Celery/Redis отсутствует в текущем окружении; legacy-структура модулей не совпадает с текущей сервисной декомпозицией.
- **Нужные изменения в текущем проекте**:
  - Добавить YouTube-аудио модуль в текущую аудио-архитектуру рядом с Redeye-провайдером.
  - Встроить вызов YouTube-обогащения в Discogs-потоки добавления/обновления записи.
  - Добавить новый массовый admin action и новую кнопку в форме записи для точечного запуска.
  - Расширить формат итоговой отчётности admin action с детализацией updated/skipped/errors.
  - Сохранить полную обратную совместимость для Redeye-путей.

## Implementation Rollout Plan

1. Зафиксировать сравнительный анализ current vs legacy как обязательный артефакт перед кодом.
2. Спроектировать и согласовать модуль YouTube-обогащения в текущем слое аудио-сервисов.
3. Встроить обогащение в поток добавления из Discogs и в admin action `update_from_discogs`.
4. Добавить отдельный admin action "Обновить аудио из YouTube" (с принудительной перезаписью) и кнопку на форме записи.
5. Реализовать единый формат результата ручных операций и политику частичных ошибок/повторов.
6. Провести регрессионную проверку существующих Redeye-потоков и подтвердить критерии успеха.
